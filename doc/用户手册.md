## 一、工具介绍

### 1.1 这是什么

**智能单元测试工具**是一个面向 C 与 Python 源码工程的自动化单元测试平台。它读取你上传（或在共享卷中）的源码工程，**自动解析出可测函数，借助大语言模型（LLM）为每个函数生成单元测试代码，编译/解释执行并采集覆盖率**，最终输出每个函数的测试结果、生成的测试代码与测试报告。

它既可独立运行，也可作为 UniPortal 工具链中的一环，与上游的"文档审查"和"需求追溯"工具联动，让生成的测试用例贴合真实需求。

### 1.2 核心能力

- **多语言支持**：C（基于 Unity 测试框架）与 Python（基于 pytest）。
- **自动函数识别**：按可见性与命名规则区分"必测函数"（对外接口 / 公开函数）与"可跳过函数"（静态函数 / 以 `_` 开头的私有方法）。
- **LLM 智能生成**：调用 DeepSeek 等兼容 OpenAI 接口的大模型，结合函数源码、代码图谱、整文件上下文、上游需求生成高质量测试用例；支持流式输出。
- **代码图谱可视化**（C）：基于 Joern 生成调用图（Call）、抽象语法树（AST）、控制流图（CFG）、程序依赖图（PDG）四类图片，辅助理解函数结构。
- **离线环境运行**（Python）：支持随项目打包 conda-pack 离线环境，**全程不联网**即可在工程指定 Python 版本下运行测试，彻底解决 C 扩展（numpy/pandas/scipy 等）版本绑定问题。
- **真实执行 + 覆盖率**：C 用 gcc 编译、lcov 采覆盖率；Python 用 pytest-cov 采覆盖率；行/分支/函数覆盖率一目了然。
- **需求追溯**：自动关联上游需求，在每个函数的生成页展示"关联需求"，并把需求内容注入 LLM prompt。
- **一键导出**：项目级 DOCX 测试报告、单函数 DOCX/Markdown/HTML、项目级 JSON（供下游工具消费）。
- **批量自动测试**：上传后或浏览页一键对全部必测函数执行"解析 → 生成 → 执行"全流程，支持并发。

### 1.3 工具全景流程

```
┌──────────┐   ┌────────────┐   ┌────────────┐   ┌────────────┐   ┌──────────┐
│ 1. 上传  │ → │ 2. 总览/   │ → │ 3. 生成    │ → │ 4. 执行    │ → │ 5. 导出  │
│  项目    │   │    浏览    │   │  测试用例  │   │   +覆盖率  │   │  报告    │
└──────────┘   └────────────┘   └────────────┘   └────────────┘   └──────────┘
 zip/文件夹/    函数树/源码      关联需求+图谱     编译运行         DOCX/MD/
 单文件         选择被测函数     LLM流式生成       行/分支覆盖      HTML/JSON
                                  +注释
```

页面顶部导航栏把这五步显示为一个引导式向导，未完成前置步骤的入口会被自动禁用。

### 1.4 适用场景与边界

- **适用**：C/Python 工程的单元测试自动化生成与回归、覆盖率度量、需求驱动的测试补充。
- **边界**：
  - 仅支持 C 与 Python 两种语言，不支持 C++/Java/Go 等。
  - 代码图谱图片（Call/AST/CFG/PDG）**仅 C 项目支持**；Python 项目使用轻量文本图谱（变量/调用/返回/分支），不生成图谱图片——因此在 Python 项目里这四类图片请求会返回 400，属正常行为。
  - 生成的测试依赖 LLM 能力，复杂或高度上下文相关的函数仍可能需要人工微调；工具提供"修复测试用例"闭环帮助迭代。

## 二、核心技术

### 2.1 整体架构

工具采用前后端分离架构：

| 层 | 技术 | 说明 |
| :--- | :--- | :--- |
| 前端 | Vue 3（`<script setup>`）+ Vite + Tailwind CSS + Pinia | 单页应用，5 步引导式界面；Browse/Generate 页 keep-alive 缓存 |
| 后端 | FastAPI（Python 3.10）+ Uvicorn | REST API，前缀 `/api`；同步执行，无外部消息队列 |
| 大模型 | DeepSeek（兼容 OpenAI `/chat/completions`） | 可替换为任意兼容服务（如 gpt-4o-mini、qwen-max） |
| 代码分析 | Joern CLI（C，需 Java 17）+ Python `ast` | 函数解析与代码图谱 |
| 容器 | Docker / docker-compose | 单服务，三个数据卷 |

后端服务结构：

```
app/
├── main.py                 # FastAPI 入口、健康检查、静态前端托管
├── routers/                # 路由
│   ├── project.py          # 上传、结构、函数详情、图谱、导出
│   ├── testcase.py         # 生成（流式/非流式）、注释、执行、导出
│   └── config.py           # LLM 配置读写与连通性测试
└── services/               # 业务逻辑
    ├── project_service.py  # 上传解压、扫描、meta.json、语言识别、路径解析
    ├── parser_service.py   # 函数解析、测试目标抽取、代码图谱
    ├── llm_service.py      # LLM 客户端、prompt 构造、流式生成
    ├── runner_service.py   # 编译执行、venv/conda 环境、覆盖率解析
    ├── upstream_service.py # 上游需求/追溯读取与函数匹配
    └── joern_service.py    # Joern CPG 生成与图谱图片
```

### 2.2 语言识别与测试框架

- **语言自动识别**：扫描工程内 `.c/.h/.py` 文件数量，`.py` 数 ≥ `.c` 数判定为 Python，否则为 C；两者皆无则为 `unknown`。识别结果写入项目的 `meta.json`。
- **测试框架默认值**：C → `unity`；Python → `pytest`。生成页可手动切换框架（C 支持 unity/cmockery/gtest，Python 仅 pytest）。
- **测试文件命名**：C 固定为 `test_runner.c`，Python 固定为 `test_generated.py`。
- **依赖管理识别**：存在 `pyproject.toml` 记为 `pyproject`，否则有 `requirements.txt` 记为 `pip`。

### 2.3 函数解析与代码图谱

**函数解析**（`parser_service.py`）：
- C 采用正则 + 花括号计数的启发式解析，按可见性把函数分为：**必测（must_test，非 static 的对外接口，或在头文件中声明）**、**可选静态（optional_static）**、**跳过（skipped）**。
- Python 采用标准库 `ast` 精确解析，支持模块级函数、类方法、`async` 函数；公开函数/方法进入必测，**以 `_` 开头（含 `__init__` 等魔术方法）判为私有并跳过**；`tests/` 目录下的函数不作为目标。

**代码图谱**：
- **文本图谱**（两语言通用）：内部变量与参数、返回值分析、调用关系、分支信息。
- **Joern CPG 图谱**（仅 C）：生成 `cpg.bin` 后产出 **调用图 / AST / CFG / PDG** 四类 PNG 图片。Python 不支持图片图谱（请求会返回 400）。

### 2.4 基于 LLM 的测试用例生成

- **模型与接口**：默认 DeepSeek（`deepseek-chat` / `deepseek-v4-flash`），兼容任意 OpenAI 接口服务。未配置 API Key 时进入 **Mock 模式**（生成占位测试，便于部署验证）。
- **生成上下文**（一并喂给模型）：被测函数源码切片、整文件源码、代码图谱、项目文件列表（路径）、上游关联需求、可选的失败上下文（用于修复再生成）。
- **用例数量按复杂度驱动**（C）：LOW 复杂度 1 个用例、MED 2–3 个、HIGH 3–5 个，避免冗余或不足。
- **双步生成**：
  1. **生成测试代码**（流式输出到页面）；
  2. **若工程含设计文档**（`function_design` 结构 JSON），第二步"注释生成"会对照设计文档为每个断言插入中文注释，并标注设计与代码的偏差。
- **测试意图（ISO/IEC/IEEE 29119）**：生成页可一键 AI 生成"测试目标 / 前置条件 / 预期结果"三段式意图，也可手填，用于引导生成。

### 2.5 离线运行环境（conda-pack）与在线 venv

Python 项目的运行环境是关键。工具支持两种方式：

**① conda-pack 离线环境（推荐，离线可用）**

- 用户在本地用 `conda pack` 把完整环境（含 Python 解释器、依赖、pytest/coverage）打包成 `env.tar.gz`，与源码一起放进 zip 上传。
- 上传时工具自动识别（按文件名提示 `env/environment/conda/venv`，并窥探包内是否含 `bin/` + `conda-meta/`），解压到项目下 `.conda_env`。
- 运行时**直接使用环境自带的 Python**（如 3.8.20），通过合成 `activate` 等价环境变量（`PATH`/`PYTHONHOME`/`VIRTUAL_ENV`/`CONDA_PREFIX`/`LD_LIBRARY_PATH`/版本匹配的 `site-packages`）激活，**不依赖容器宿主 Python**，从而保证 numpy/pandas/scipy 等 C 扩展 `.so` 与解释器版本严格匹配。
- 首次运行按需执行一次 `conda-unpack`（重写构建机旧前缀），随后幂等复用。

**② 在线 venv 回退（无离线环境时）**

- 工具在私有可写目录创建 `venv`，联网 `pip install pytest coverage pytest-cov`，并按 `requirements.txt`/`pyproject.toml` 安装项目依赖。
- 依赖指纹（`.venv.deps.hash`）变化才重装，跨任务复用。
- 此方式需要网络，且依赖版本可能不一致，复杂项目（含 C 扩展）建议优先用 conda-pack。

> **共享卷项目（集成模式）特殊处理**：共享卷只读，且项目不经上传流程、没有 `meta.json`。工具会在运行时自动发现源码目录里的 `env.tar.gz`（或已解压的 `.conda_env`），落盘到私有可写缓存 `LOCAL_WORKSPACES_DIR/{project_id}/.conda_env` 再激活，并用目录锁保证并发只落一次。详见 `doc/离线环境打包指南.md`。

### 2.6 编译执行与覆盖率采集

- **C**：拷入 Unity 框架文件 → 去掉被测源码前缀 `static` 以便链接 → `gcc -fprofile-arcs -ftest-coverage` 编译被测文件 → 用 `objcopy --weaken-symbol` 把除被测函数外的符号弱化（从二进制层消除"multiple definition"冲突）→ 链接 `--coverage -lm -lz` → 运行（10 秒超时）→ 解析 Unity 的 `:PASS/:FAIL/:IGNORE` → `lcov` 采集行/分支覆盖率。
- **Python**：在被测目录运行 `pytest-cov`（`--cov --cov-branch --cov-report=json`）→ 解析用例通过/失败/跳过与 `coverage.json` 覆盖率（超时默认 300 秒）。
- **依赖预检**：执行前先尝试 `import 被测模块`，若发生 `ModuleNotFoundError` 立即短路并给出清晰提示（区别于"被测模块顶层副作用"抛出的其它异常，避免误报）。

### 2.7 上游工具链集成（需求追溯）

工具可读取同一工程下上游工具的输出，为测试注入需求上下文：

| 上游工具 | 目录 | 文件 | 作用 |
| :--- | :--- | :--- | :--- |
| 文档审查 | `document-validator/` | `requirements.json` 或 `requirement.json` | 需求列表（标题/正文/表格） |
| 需求追溯 | `traceability_link_recovery/` | `traceability.json` | 需求↔函数映射（取 `src_rank=1` 最佳匹配） |

- **总览状态**：在生成页顶部横幅展示"已关联 N 条需求 / 追溯链路已匹配 N 条链接"。
- **单函数关联需求**：按函数名（兼容裸名、`Class.method`、`module.Class.method` 末段等多种形式）+ 源文件 basename 匹配，取最佳需求并在生成页展示需求标识/标题/内容/表格。
- **注入 LLM**：生成用例时把匹配到的需求标题、正文、表格约束追加进 prompt，引导覆盖需求场景与边界。
- 两类上游输出均缺失时，按常规逻辑生成，无影响。

### 2.8 两种部署/集成模式

由环境变量 `UNIPORTAL_STORAGE_PATH` 是否设置决定：

| 模式 | 触发条件 | 源码来源 | 写入位置 |
| :--- | :--- | :--- | :--- |
| **独立模式** | 未设置 | 本地 `workspaces/`（上传写入） | 同一 `workspaces/` |
| **集成模式（UniPortal）** | 已设置（如 `/data/uniportal`） | 只读共享卷 `{portal_project_id}/{item_id}/...` | 私有可写卷 `LOCAL_WORKSPACES_DIR` |

集成模式下：项目列表按 `portal_project_id`（由 iframe URL query 传入）过滤；工具生成物（CPG、图谱、缓存、`.conda_env`）只写私有卷；导出 JSON 会回写到共享卷的 `unit-test-generate/unit-test-generate.json` 供下游消费。

## 三、操作步骤

### 3.1 部署与首次配置

**Docker 部署（推荐）**

```bash
docker-compose up --build -d
```

- 默认对外端口 **8004**（映射容器 8000）。访问 `http://<服务器IP>:8004`。
- 三个数据卷：`uniportal_storage`（只读共享卷，集成模式用，需 UniPortal 先创建）、`tool_local`（私有可写，存 CPG/图谱/缓存/`.conda_env`）、`tool_tasks`（任务沙盒）。
- 配置 LLM Key：编辑 `.env` 写入 `DEEPSEEK_API_KEY=sk-xxxx`，或在页面右上角齿轮按钮打开"LLM 接入配置"在线配置（见 [3.8](#38-llm-接入配置)）。未配置时为 Mock 模式。

**裸机部署**：后端 `uvicorn app.main:app --host 0.0.0.0 --port 8000`；前端 `cd Frontend && npm run dev`（开发端口 5173）。详见 `DOCKER.md` / `start.sh`。

> 修改代码后重建容器：`docker-compose down && docker-compose up --build -d`。

### 3.2 第一步：上传项目

进入首页（**项目上传**）。支持三种上传方式，均进入 `POST /api/project/upload`：

1. **单文件 / zip**：点击或拖拽，支持 `.c / .h / .py / .zip`。
2. **整个文件夹**：点击"或选择整个文件夹（自动打包为 zip）"，前端用 JSZip 在浏览器内打包（含 `env.tar.gz`）后上传，适合含离线环境的大目录。
3. **拖拽**：直接拖文件到上传区。

填写"项目名称"（可留空，自动取文件名），点击 **"确认上传并开始分析"**。

**Python 项目离线环境打包（重要）**

若 Python 项目含第三方依赖（尤其 C 扩展），需同时上传打包好的运行环境。点击上传区 **"如何打包 env.tar.gz？"** 链接查看弹窗指引，核心命令：

```bash
conda create -n myenv python=3.8
conda activate myenv
pip install -r requirements.txt          # 项目依赖
pip install pytest coverage pytest-cov    # 测试工具（必须）
conda pack -n myenv -o env.tar.gz --force --ignore-missing-files
```

然后把 `env.tar.gz` 与源码放进同一个 zip 上传（或直接上传整个文件夹）。详见 `doc/离线环境打包指南.md`。

> 上传后可选择勾选 **"上传后自动对必测函数执行全量测试"**（见 [3.7](#37-批量自动测试)），跳过手动逐步操作，直接得到全量结果。

### 3.3 第二步：总览看板与项目浏览

上传成功（或从首页"已有项目"选择）后进入 **总览看板**（`/dashboard`）：

- 顶部统计卡片：总被测函数 / 通过 / 失败 / 编译错误数量。
- 主表格：每个函数的名称、源文件、行覆盖率、分支覆盖率、状态、操作（查看详情 / 查看源码）。
- 右上角可 **导出 DOCX 测试报告** / **导出 JSON**。

点击 **"进入项目浏览"** 进入 **项目浏览**（`/browse`），三栏布局：

- **左栏 项目结构**：文件树；JSON 设计文档会标"设计文档"角标；每个源文件节点下展开其函数列表。
- **中栏 源码视图**：语法高亮显示选中文件；点击函数名（或函数体内某一行）即选中该函数并定位到对应行。
- **右下角 浮动按钮**：选中函数后出现 **"生成测试用例"**，点击进入生成页。

> 也可以在左栏点闪电图标 **"执行全量自动测试"** 一键批量测试（见 [3.7](#37-批量自动测试)）。

### 3.4 第三步：生成测试用例

进入 **测试生成**（`/generate`），左右两栏：

**左栏（函数分析）**

- **上游状态横幅**：显示需求/追溯链路匹配情况。
- **函数基本信息**：签名、函数名、所在行号。
- **关联需求**：展示该函数匹配到的需求（相似度、需求标识/标题/内容/表格）。这是需求驱动的关键卡片。
- **代码图谱**：变量与参数、返回值分析；C 项目额外展示 Call/AST/CFG/PDG 四张图（每张可单独刷新）。Python 仅有文本图谱。
- **函数意图**：可手填或点 **"AI 辅助生成"** 生成 ISO 29119 三段式意图。
- **框架选择**：按语言选择测试框架，点击 **"立即生成测试用例"**。

**右栏（生成代码）**

- LLM **流式输出**测试代码，实时呈现。
- 生成完成后，若工程含设计文档，会自动进入第二步"插入设计文档注释"。
- 提供 **复制代码** 与绿色 **"编译并执行"** 按钮。

### 3.5 第四步：执行与查看结果

点击"编译并执行"进入 **执行结果**（`/result`），自动运行测试：

- **状态头**：环形进度 + 通过/失败/总数；语言相关的状态文案（C"正在编译测试用例"、Python"正在准备 pytest 测试环境"）。
- **控制台输出**：终端样式，PASS 绿 / FAIL 红，含 stderr 错误日志。
- **覆盖率分析**：行覆盖率 / 函数覆盖率 / 分支覆盖率三条进度条（>80% 绿、>50% 黄、否则红）。
- **被测函数详情**：源码逐行染色，覆盖行绿色、未覆盖行红色，直观看出未测分支。
- **操作按钮**：
  - **"修复测试用例"**：编译失败或有失败用例时出现，携带失败上下文回到生成页让 LLM 修复，形成闭环。
  - **"重新执行"**：不重新生成，直接再跑一次。
  - **"导出 DOCX" / "MD"**：导出该函数的报告。

### 3.6 第五步：导出报告

| 导出内容 | 入口 | 格式 | 说明 |
| :--- | :--- | :--- | :--- |
| 项目测试报告 | 总览看板"导出 DOCX" | DOCX | 含总体统计表 + 每个函数的详情（源文件/签名/状态/覆盖率）+ 测试代码（按用例拆分，英文注释转可读中文段落） |
| 项目结果汇总 | 总览看板"导出 JSON" | JSON | 全部函数状态/覆盖率/测试代码/错误；集成模式额外回写共享卷 |
| 单函数报告 | 执行结果页"导出 DOCX/MD" | DOCX/MD/HTML | 单函数的基本信息、源码、设计文档、用例、结果、覆盖率 |

### 3.7 批量自动测试

适用于一次性得到整个工程的测试结果。两种入口：

- **上传时**：勾选"上传后自动对必测函数执行全量测试"，并选择并发数（2/4/8/16）。
- **浏览时**：左栏闪电按钮打开批量测试浮层。

流程均为 **解析 → 生成 → 执行** 三阶段，实时显示进度与 成功/失败/编译错 统计，可中途 **"停止测试"**；完成后"前往结果总览"查看。仅对必测函数（外部接口/公开函数）执行，私有函数为间接测试。

### 3.8 LLM 接入配置

页面右上角齿轮按钮 → **"LLM 接入配置"**：

- **Base URL**：默认 `https://api.deepseek.com`，可换任意 OpenAI 兼容服务。
- **API Key**：DeepSeek / OpenAI 等。
- **Model**：如 `deepseek-chat`、`gpt-4o-mini`、`qwen-max`。
- 点 **"测试连接"** 验证可用性后保存。配置写入 `.env` 持久化。

> 推理类/flash 模型会被自动识别并放宽 `max_tokens`（≥16000）与超时（300s），避免输出被截断。

### 3.9 常见问题（FAQ）

**Q1：Python 项目报 `No module named 'pandas'`（或其它第三方库）？**
缺少离线运行环境。请按 [3.2](#32-第一步上传项目) 用 `conda pack` 打包 `env.tar.gz` 随源码上传；或确保能联网（工具会回退在线 venv 并按 `requirements.txt` 安装）。含 C 扩展的库（numpy/pandas/scipy）必须用 conda-pack，版本才能对上。

**Q2：Python 项目里 Call/AST/CFG/PDG 图谱报 400？**
正常。图谱图片仅 C 项目支持，Python 用文本图谱。忽略这些 400 即可。

**Q3：共享卷（集成模式）项目不显示"关联需求"？**
已在最新版本修复：切换函数时会重新拉取需求上下文。若仍异常，确认工程目录下存在 `document-validator/requirements.json` 与 `traceability_link_recovery/traceability.json`，且 `traceability.json` 中有对应函数名的链路。

**Q4：生成的是占位测试 / Mock 代码？**
未配置有效 LLM API Key，进入 Mock 模式。按 [3.8](#38-llm-接入配置) 配置 Key。

**Q5：编译失败 / 用例不通过怎么办？**
在执行结果页点 **"修复测试用例"**，工具会把失败上下文带回生成页让 LLM 修正，可多轮迭代。

**Q6：私有函数（`_` 开头 / `static`）没有被测试？**
符合设计：Python 以 `_` 开头、C 的 `static` 函数默认视为内部实现、间接测试，不计入必测目标。如需测试，可在生成页手动定位或调整源码可见性。

**Q7：集成模式无法删除项目？**
UniPortal 项目由门户统一管理，工具内不允许删除（会返回 403）；请在 UniPortal 侧管理。仅"本地私有上传"的项目可在工具内删除。
